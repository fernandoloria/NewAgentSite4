# API Migration Plan

> Documento generado por análisis estático inicial del WebForms. No se modificó código fuente de la aplicación; solo se agregan documentos de planificación. Donde el análisis estático no permite confirmar reglas de negocio, parámetros exactos, tablas o semántica de SP, se marca como **pendiente de confirmar**.

## 1. Executive Summary

La aplicación contiene **104 pantallas en `Report`**, **19 popups en `Popup`** y **20 handlers en `Services`**. Se detectó uso intensivo de `Session`, permisos `Common.HasRights`/`Security.AgentRights`, `QueryString`, controles WebForms, SQL Server, stored procedures y handlers ASHX que ya devuelven JSON parcial pero sin contrato REST homogéneo. La migración recomendada es **híbrida por fases**: crear una API separada, mantener WebForms operativo, exponer primero consultas simples, luego reportes complejos, luego operaciones transaccionales y finalmente deletes/procesos críticos.

## 2. Current WebForms Application Overview

| Área | Hallazgo | Implicación API |
|---|---|---|
| Módulos principales | Reports, Popup, Services, Controls/Login, BasePage, AgentLines | Crear controllers por dominio y no por página cuando sea posible |
| Autenticación | Login guarda `Authenticated`, `userdata`, `IdAgent`, `SubIdAgent`, `Agent`, `SubAgent`, `IsDistributor`, `AgentRights`, `SubAgentRights`, `IdCurrency`, cultura e IP en Session | Migrar a JWT + refresh token + endpoint de contexto |
| Permisos | `Common.HasRights(ReportPosition.*)` y `Security.AgentRights.Has(...)` | Claims/permissions y policies por endpoint |
| Estado | Session y ViewState sostienen filtros, jerarquía y contexto visual | Convertir a DTOs explícitos y cache controlado |
| Datos | SQL Server por SP y SQL directo; tablas Agent, Player, Language y otras pendientes | Capa DAL/repository con parámetros tipados |
| Riesgos | Escrituras en reportes/popups/services, SP sin documentación, JS antiguo, falta de pruebas | Migración con comparación WebForms vs API y auditoría |

## 8. JWT Security Plan

| Elemento | Propuesta |
|---|---|
| Login | `POST /api/auth/login` valida credenciales actuales y emite access token corto + refresh token |
| Claims | `userId`, `username`, `agent`, `idAgent`, `subIdAgent`, `subAgent`, `isDistributor`, `idCurrency`, `idLanguage`, `culture`, `ip`, `permissions`, `agentRights`, `subAgentRights`, `timezone` |
| Contexto jerárquico | Reemplazar `ChangeSubAgent.ashx` por `POST /api/security/context/sub-agent`; no confiar solo en claims antiguos para cambio de subagente |
| Expiración | Access token 15-30 min; refresh token rotativo 8-12 h o política del negocio |
| Autorización | Policies por `ReportPosition`, por derechos como `DELETEBETS`, y ownership Agent/Player en DAL |
| Riesgos | QueryString con ids sensibles, deletes de wagers, pagos/free play, límites, mensajes y perfiles requieren auditoría |

## 11. Recommended API Architecture

| Opción | Ventajas | Desventajas/Riesgos | Tiempo | Recomendación |
|---|---|---|---|---|
| A. Web API dentro del mismo WebForms | Rápido, comparte Session/config | Acopla legacy, dificulta JWT moderno, riesgo de romper app | Bajo inicial/alto largo plazo | Solo puente temporal |
| B. Nuevo ASP.NET Web API 2 .NET 4.8 | Compatible con assemblies legacy y SP, JWT viable, menor riesgo | Sigue en Framework, menos moderno que Core | Medio | Buena fase 1 |
| C. Nueva ASP.NET Core | Mejor seguridad, DI, Swagger, hosting y Angular | Reutilización legacy limitada, requiere adaptar DAL | Medio/Alto | Meta estratégica |
| D. Híbrida por fases | Mantiene WebForms, permite Web API 2 o Core por módulos, reduce riesgo | Requiere disciplina de contratos y comparación | Medio | **Recomendada** |

## 15. Technical Risks

Lógica de negocio en code-behind, dependencia de Session/ViewState, QueryString inseguro, SQL directo mezclado con UI, SP no documentados, escrituras en popups/services, handlers usados por JS antiguo, falta de pruebas, diferencias WebForms/API, performance de reportes grandes, necesidad de auditoría, permisos no centralizados y operaciones financieras/de borrado con alto impacto.

## 16. Acceptance Criteria

Todos los reportes tienen endpoint equivalente o justificación; todos los popups/services documentados; operaciones SELECT/INSERT/UPDATE/DELETE identificadas; endpoints protegidos por JWT y permisos; JSON y errores estándar; Swagger completo; comparación contra WebForms; pruebas de escritura válidas/ inválidas; Angular PWA puede consumir auth, catálogos, filtros, reportes y transacciones.
